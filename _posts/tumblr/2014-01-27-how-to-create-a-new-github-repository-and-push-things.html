I use shell to interact with my Github repositories because I’m using a Linux machine. 

So, first thing you should do is install git onto your machine, you can use: 

sudo apt-get install git

Alright, so back to how to create a repository & be able to push things to it.

1. Create a repository using the github.com site. 

Creating your repository, you’ll end up with a screen like: 

I’m just going to fill it out like so. But be aware that it’s very important what you call your repository. So create the repository: 

So in that bottom-right corner, you’ll see something that says "HTTPS clone URL”. Using that, you can clone your repository into your local file system (or any server you SSH into, as long as that server has git). What is cloning? Your repository is essentially a folder with files in it. When you clone a repository, you’re taking that folder and downloading it into your own file system. 

2. Clone your repository and initialize it.

git clone {YOUR HTTPS URL}

With the “git clone” command, you can clone your repository into your file system. Replace {YOUR HTTPS URL} with the HTTPS clone URL you saw on the bottom right corner of your repository on the github.com webpage (should have a .git suffix). When you do that, you should see your repository in the folder you git clone’d in. The name of that folder should match your repository name (which is why I said the name will be important).

Go ahead and open up that folder you cloned and its contents should match the ones in your github repository that you see on the website. So, for my test-repo, I should see a README.md. Now you should initialize your repository. Think of initializing as preparing your now local folder for interacting with github. If you don’t do this, you can’t update your repository. Of course, you’ll be able to update your local folder, but the changes won’t be seen online. You can initialize this by typing this into the shell (while you’re cd’d into your repository folder):

git init

3. Add files, commit, and push.

So now you can add files to your local repository folder. You can use vim, notepad(++), an IDE, any editor. As long as you save those files in that repository folder. 

Now that you have files, let’s add them all to your files-I-want-committed list. 

Right now, these are the files in my test-repo repository folder that I have saved locally. I want to add all these files to my files-I-want-committed list. 

Before we add these files to that list, let’s first check if our list is empty or not. Do this by: 

git status

Right now, since I haven’t added any files to this list, I get a message that says: 

Basically, all the files I want to add right now are “untracked”, meaning they are NOT added to the files-I-want-committed list. As you can tell from the last line, you can “track” files by using “git add {FILEPATH}”. Note that the {FILEPATH} does not need to be the complete filepath from the root, just from the local repository directory. 

But in this case, since we just want to add ALL the files, you can just use: 

git add *

Note that the asterisk means “all files in that folder”. This will then make all of your files “tracked” or added to the files-I-want-committed list (they are synonymous). 

Of course, if you wanted to track specific files, you’d have to specify the filename after the git add. For example, I only want to track one.txt, two.txt, and four.txt because I don’t want to yet change three.txt in my github repository for whatever reason. To do this, I’d need to type in:

git add one.txt
git add two.txt
git add dir1/four.txt

Now that we tracked a bunch of files, we want to commit them now. If I ask for the git status right now, (using the previous git add sequence): 

I usually do a git status after every couple git add statements to make sure all of my files are added, but that’s just me being paranoid. 

Now you’re ready to commit. Committing on github is how github manages your versions. You can track your file versions using your commits. That is why github allows you to commit with a commit message. The short way I include messages for github (I know there are more ways to do this) is: 

git commit -m "my first commit, changed one.txt, two.txt, four.txt"

Ideally, your commit messages should be useful. I’d like to think of them as tweets (they should be as concise as possible). Or, if you’re more of a writer, theses statements (jk, that’s just scary).

After this, you’re ready to push. OR, you could commit more things THEN push it. I went ahead and added a “five.txt” and committed it. 

To push commits means to make them available to everyone who looks at your git repository. This is also useful if you’re working on a group project and you’re all accessing this one repository. None of your team members are going to see your commits/changes unless you push your commits. 

git push origin master

After this, git is going to prompt you for your username and password for github. Enter that in and your push should go through.

After this, you should see it on your repository on the github webpage.

And if you want to see your commits on the webpage, go to the link that says “{NUMBER} Commits” above this table.

Remember how I added that extra five.txt file? I also committed that “first commit” beforehand. I didn’t push in between the two commits, but the separate commits still show up. This has been very helpful when I didn’t feel like pushing after every edit I made, so you don’t have to worry about pushing all the time, until the end of the day or something. 

Hopefully this was helpful, there are way more ways to do this simple process, but this is just how I do it. If I find any better ways, I will share in tutorial form here.
